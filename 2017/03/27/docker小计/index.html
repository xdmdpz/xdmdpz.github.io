Docker Engine

Docker Engine是Docker的大脑，docker指令是通过Engine运行，Engin的底层依赖Linux内核，也就是说Engin只能安装在Linux的各个发行版上，Mac OS 和 Windows 上无法直接运行Engine，需要借助一个装有lunux系统的虚拟机来。这个虚拟机被成之内Docker Machine(boot2dicker)

Docker Machine

Docker Machine 以主机的形态封装了装有Docker Engine的虚拟机实例，Machine的出现是为了替代boot2docker，以统一全操作系统上的Docker管理。Machine的本地底层Driver是Ocacle VirtualBox，云提供商AWS、Microsoft Azure、Google Compute Engine和Digital Ocean等也提供了对Docker Machine的支持。相对于Docker Swarm而言，Machine创建的虚拟机实例是Docker集群上的一个节点（Node）。

Docker Swarm

Docker Swarm用于管理集群上的Node以及容器调度。Swarm的出现，渗透着Docker一同容器圈的野心。对于开发者来说，这一趋势的好处就是不用额外找解决方案，只需跟进docker自身的发展，就基本上够用了，坏处就是呗绑死在一家厂商身上，如果Docker产品线能解决的问题有局限，我们就很难跳出来了。

Docker Compose

Docker Compose 的前身是 fig，用于编排多个容器。其定位是为了开发者提供开发和测试环境，以节省时间、提高效率。Compose 不适用与大型生产环境的部署。

Docker Toolbox

Docker Toolbox 是为了MAC os和Windows操作系统提供的一个工具箱，里面包含了Engine、Machine、Compose和镜像图形管理工具Kitematic。安装Docker Toolbox后。启动菜单就会多出两个图标、分别是Docker Quickstart Terminal 、Kitematic

Docker image

Remote docker hub

集中存储镜像的Web服务器地址。该部分的存在是的可以区分从不同镜像库中拉去的镜像，若docker的镜像中缺少该部分，说明使用的是默认镜像库，即Docker官方镜像库

NameSpace

类似于github中的命名空间，是一个用户或组织中所有镜像的集合。

Repository

类似于Git仓库，一个仓库可以用多个镜像，不同的镜像通过tag来区分。

Tag

类似与Git仓库里面的tag，一般用来区分同一类镜像的不同版本

layer

镜像由一系列层组层，每层都用64位的十六进制数标示，非常类似于Git仓库中的commit

image ID

镜像最上层的layer ID就是该镜像的ID，Repo:tag提供了易于人类是别的名字，而ID便于脚本处理、操作镜像
